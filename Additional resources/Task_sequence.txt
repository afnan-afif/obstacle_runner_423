## 6. GITHUB & TASK COMPLETION SEQUENCE
Since the entire project must reside in a single `.py` file, the team MUST follow this strict sequence to avoid Git merge conflicts. No one is allowed to push code outside their designated Zone.

**Phase 0: Initialization & Zoning (Member 1)**
- Member 1 creates the GitHub repository with the base template.
- Member 1 adds the empty Zone comment blocks (e.g., `# === MEMBER 2: PLAYER ===`) to establish boundaries.
- Member 1 pushes this foundation to the `main` branch. 

**Phase 1: The Environment (Member 1)**
- Member 1 creates branch `feature-world`.
- Codes the scrolling floor grid and camera setup strictly inside the "World" zone.
- Merges `feature-world` into `main`. (The game now has an endless floor and camera).

**Phase 2: The Player (Member 2)**
- Member 2 pulls the updated `main` branch and creates branch `feature-player`.
- Codes the player cube, lane switching, jumping, and rolling states inside the "Player" zone.
- Merges `feature-player` into `main`. (The game now has a floor and a controllable player).

**Phase 3: The Threat (Member 3)**
- Member 3 pulls the updated `main` branch and creates branch `feature-obstacles`.
- Codes the obstacle manager and AABB collision detection inside the "Obstacles" zone.
- Merges `feature-obstacles` into `main`. (The core game loop is now fully playable).

**Phase 4: Novelty Integration (Parallel)**
- With the core engine stable and zones respected, all members pull the latest `main`.
- Everyone branches off simultaneously (`feature-warp`, `feature-gravity`, `feature-morph`).
- Members code their specific novelty features inside their designated zones.
- Branches are merged back into `main` one by one. Conflicts are avoided because all logic is contained within separate zones.
